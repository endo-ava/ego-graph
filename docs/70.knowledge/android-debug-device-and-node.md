# Android 常設デバッグ端末運用と Android Node 構想

このドキュメントは、EgoGraph における Android 実機の常設運用方針を整理したものです。
主目的は、エージェントがいつでも利用できるデバッグ専用端末を持つことです。
その発展案として、Android を音声・カメラ・センサーを持つ `Android Node` として扱う構想もあわせて整理します。

## 結論

- まず優先すべきは、`デバッグ専用 Android 端末の常設運用`
- 無線 ADB と Tailscale を組み合わせることで、`いつでも再接続しやすい端末` は実現可能
- ただし、`完全に切れない常時接続` は前提にせず、`自動再接続前提の安定運用` として設計する
- 将来的には、その常設端末を発展させて `Android Node` として扱うのが自然

## なぜ常設デバッグ端末が必要か

現状の課題は、Android 操作や実機確認のたびに以下の準備が必要になりやすいことです。

- デスクトップ PC 側でエミュレーターを起動する
- ADB 接続先の準備をする
- 実機確認のたびに人が環境を整える

これでは、エージェントに継続的なデバッグを任せにくくなります。

常設デバッグ端末を 1 台用意すると、次の状態に近づけます。

- エージェントが必要なときに `adb` で接続確認できる
- APK のインストール、起動、ログ確認、スクリーンショット取得を即実行できる
- エミュレーター起動待ちなしで実機確認できる
- Maestro や簡易 E2E の起点として使える

## 基本方針

常設運用の主役は `デバッグ専用 Android 端末` です。
`Android Node` はその延長線上にある発展版として扱います。

優先順位:

1. 安定して再接続できるデバッグ端末を作る
2. 端末状態を保守しやすい運用にする
3. 将来的にセンサー・音声・カメラ利用へ拡張する

## 常設デバッグ端末の運用方針

### 目標

- エージェントが任意のタイミングで ADB 接続確認できる
- 実機ビルド、インストール、起動、ログ取得を自動で回せる
- 端末状態が壊れても復旧経路が明確である

### 非目標

- 完全無停止の常時接続保証
- 人間と同等の自由な UI 操作
- 初回権限許可や OEM 固有画面の完全自動化

### 推奨構成

- 余剰 Android 端末を 1 台、完全にデバッグ専用化する
- その端末を固定の Wi-Fi 環境で運用する
- Tailscale を利用して、端末または中継ホストへの到達性を安定化する
- ADB は `tcpip 5555` など固定ポートでの再接続を基本にする
- 再接続は手動前提にせず、監視スクリプトや tmux/systemd で補助する

### ネットワーク設計

Tailscale を使えるなら、課題はかなり減ります。
重要なのは、`ポート固定` だけでなく `到達先のアドレス固定` も実現することです。

必要条件:

- 端末側の ADB 待受ポートを固定する
- 到達先 IP が変わらないようにする
- 同一 Tailnet 内から安定して到達できるようにする

現実的な候補は次の 2 つです。

#### 1. Android 端末に直接到達する構成

- Android 端末側で Tailscale を利用
- `adb tcpip 5555` で待受ポートを固定
- Linux 側から `adb connect <tailscale-ip>:5555`

利点:

- 構成がシンプル
- 中継ホストなしで端末へ直接到達できる

注意点:

- Android 側で Tailscale や Wi-Fi が落ちるとそのまま切れる
- 端末再起動後の状態維持は機種差・OS 制約の影響を受けやすい

#### 2. 常時稼働ホスト経由で到達する構成

- Android 端末は USB またはローカル Wi-Fi で常時稼働ホストへ接続
- そのホストに Tailscale を入れる
- Linux 側から中継ホストへ入り、そこから `adb connect` や `adb -s` を使う

利点:

- 端末側のネットワーク変動を吸収しやすい
- 自動再接続や監視をホスト側に集約しやすい

注意点:

- 構成は少し複雑になる
- 中継ホストの保守が増える

安定性を重視するなら、まずは `常時稼働ホスト経由` の方が運用しやすいです。
シンプルさを優先するなら `Android 端末へ直接到達` でも十分始められます。

### 端末設定の推奨事項

常設運用では、端末を「普段使いスマホ」と切り離すことが重要です。

- 画面ロックなし、または極力軽い解除方式にする
- スリープ時間を長めにする
- 開発者向けオプションを有効にする
- USB デバッグを有効にする
- 不要な通知や自動最適化を切る
- 自動 OS 更新、自動アプリ更新を極力避ける
- バッテリー最適化の影響を受けるアプリは除外を検討する
- 専用 Google アカウントまたは最小構成で運用する

### エージェントに任せやすい操作

常設デバッグ端末でエージェントに任せやすいのは、主に次の範囲です。

- APK のインストール
- アプリ起動
- `logcat` 取得
- スクリーンショット取得
- 画面タップやスワイプなどの簡易操作
- プロセス状態、アクティビティ状態の確認
- Maestro による定型的な画面遷移確認

逆に、次は詰まりやすいです。

- 初回権限ダイアログの複雑な分岐
- 指紋認証や顔認証
- OEM 独自設定画面
- ロック解除
- カメラや通知まわりの長時間バックグラウンド検証

### 運用の実態

「常時接続」というより、実際には次の状態を目指します。

- 端末はほぼ常にオンライン
- ADB は切れてもすぐ再接続できる
- 再接続の手順はエージェントが実行可能
- だめなときだけ人が介入する

この考え方にすると、安定性と運用コストのバランスが取りやすくなります。

## 推奨する最小運用

最初の 1 台は、やりすぎない構成がよいです。

### 最小構成

- デバッグ専用 Android 端末 1 台
- 固定した Wi-Fi 環境
- Tailscale 導入
- ADB の固定ポート化
- 再接続確認用のスクリプト
- アプリのインストール、起動、ログ確認の定型コマンド

### 最初に自動化したいこと

- `adb connect` の自動再試行
- 端末接続確認
- ビルド後の APK インストール
- アプリ起動
- `logcat` の採取
- スクリーンショット保存

### 運用ルール

- この端末は日常利用しない
- 不要なアプリを入れない
- OS や Play 系更新のタイミングを管理する
- 端末状態が壊れたらすぐ初期化できるようにしておく

## このプロジェクトとの相性

EgoGraph では、すでに以下の方向性があります。

- Frontend は Kotlin Multiplatform / Compose Multiplatform
- Backend は API とツール呼び出しの調停役を担える
- Gateway は端末やセッションとの接続管理に寄せやすい
- `adb connect` を使ったデバッグ前提の手順が既にある

そのため、常設デバッグ端末は単発の便利機能ではなく、開発基盤の一部として組み込みやすいです。

## 発展版: Android Node 構想

常設デバッグ端末が安定してきたら、その次の発展として `Android Node` を考えられます。

ここでいう `Android Node` とは、Android を単なる UI 実行先ではなく、エージェントが利用できるセンサー・アクチュエータ付きノードとして扱う考え方です。

### コンセプト

Android は次の能力を持っています。

- マイク
- カメラ
- 位置情報
- 通知
- バッテリー情報
- ネットワーク情報
- Bluetooth / 各種センサー

この能力を、エージェントが必要に応じて利用できる形にするのが `Android Node` です。

イメージ:

- `Debug Device`: 実機確認のための端末
- `Android Node`: 現実世界を観測し、必要時に行動できる端末

### なぜ段階を分けるべきか

いきなり Android Node を実装すると、論点が増えすぎます。

- バックグラウンド実行制約
- 権限設計
- プライバシー設計
- 音声・画像データの扱い
- 端末側常駐処理
- リモート命令実行の安全性

そのため、まずは `デバッグ専用端末` として運用し、
その上で責務を広げるのが安全です。

## Android Node でできること

将来的には、次のような機能が考えられます。

- 写真を 1 枚撮影してサーバーへ送る
- 数秒だけ音声を録音して送る
- 通知イベントを収集する
- 位置情報変化をイベント化する
- 充電開始や Wi-Fi 接続をトリガーに処理を走らせる
- 端末状態を heartbeat として定期送信する

これは「エージェントが Android を所有する」という感覚に近いですが、
実装上は `ユーザーが明示的に許可した範囲で利用できる専用ノード` として捉えるべきです。

## Android Node の責務分離

発展版では、`adb` を主役にしないことが重要です。

- `adb`: デバッグ、保守、復旧、アプリ更新
- Android アプリ本体: センサー取得、撮影、録音、イベント送信
- Backend: オーケストレーション、認可、ジョブ管理
- Gateway: 接続管理、通知や push 系の仲介

つまり、Node 化の本命は `専用 Android アプリ` です。
`adb` はあくまで保守チャネルとして残すのがよいです。

## Android Node の MVP

最初の MVP は、監視色を強くしすぎない方がよいです。

### 推奨 MVP

- Node のオンライン状態を返す heartbeat
- バッテリー、ネットワーク、アプリバージョンの送信
- 手動トリガーで写真を 1 枚撮る
- 手動トリガーで短い録音を 1 本取る
- 取得物を Backend にアップロードする

### まだ入れない方がよいもの

- 常時録音
- 常時カメラ
- 長時間バックグラウンド監視
- 自動判断での継続撮影
- 権限の広い収集

## 想定アーキテクチャ

### Phase 1: Debug Device

- Android 実機をデバッグ専用で常設
- ADB とビルド配布を安定化
- 実機確認をエージェントが繰り返せるようにする

### Phase 2: Managed Device

- Android アプリが heartbeat を返す
- 端末状態を Backend / Gateway から見えるようにする
- デバッグ端末から「管理対象デバイス」へ昇格させる

### Phase 3: Android Node

- 写真、録音、通知、センサーなどのワンショット取得
- トリガーベースのイベント送信
- LLM / Agent からの指示で必要な観測を実行

## リスクと注意事項

### 技術的リスク

- ADB over Wi-Fi は切断されうる
- Android のバックグラウンド制約は厳しい
- 機種依存や OEM 依存がある
- 実機 1 台だけでは端末差分を吸収できない

### セキュリティとプライバシー

- 音声・画像・位置情報は高リスクデータ
- 常設ノード化するほど、侵害時の影響が大きくなる
- 利用可能な機能は明示的な許可と可視ログを前提にすべき
- デバッグ端末と個人用端末は分離すべき

## 推奨ロードマップ

1. デバッグ専用端末 1 台を常設化する
2. Tailscale と ADB 再接続を安定化する
3. エージェントが APK 配布、起動、ログ取得を自動で回せるようにする
4. heartbeat を返す Android 側機能を追加する
5. 写真 1 枚、短い録音 1 本の取得を MVP として試す
6. `Android Node` として責務を拡張する

## まとめ

本プロジェクトでは、最初から Android Node を目指すよりも、
まず `エージェントが自由に使える常設デバッグ端末` を作る方が価値が高いです。

この段階で得られる価値は大きいです。

- 実機確認の待ち時間を減らせる
- エージェントに繰り返しデバッグを任せやすくなる
- 将来の Android Node 化にそのままつながる

その上で、発展版として Android をセンサーノード化すれば、
EgoGraph にとって「現実世界へのインターフェース」を持つ強い拡張になります。
